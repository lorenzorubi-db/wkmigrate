# Usage Guide

wkmigrate is a Python library which automates migrations from Azure Data Factory pipelines to Databricks Lakeflow jobs. This guide walks through an end-to-end pipeline migration using wkmigrate.

<Admonition type="info" title="Prerequisites">
You must have the following to run this guide:

- A working Python environment with wkmigrate installed (e.g. using `pip install wkmigrate`).
- Network access from where you run this code to Azure Data Factory and Databricks.
- An Azure Service Principal with Data Factory Contributor permissions in Azure Data Factory
- A Databricks Service Principal or personal access token

If you already have ADF pipeline JSON files locally, see [Working with Exported Pipelines](./json-pipelines) for an alternative that does not require Azure credentials.
</Admonition>

## Configure a Factory Definition Store

Create a `FactoryDefinitionStore` to load pipeline definitions from Azure Data Factory. Instantiate the
definition store using your credentials, resource group name, and Azure Data Factory resource name.

```python
from wkmigrate.definition_stores.factory_definition_store import FactoryDefinitionStore

# Specify credentials and service details of the Azure Data Factory resource
factory_options = {
    "tenant_id": "<YOUR AZURE TENANT ID>",
    "client_id": "<YOUR SERVICE PRINCIPAL CLIENT ID>",
    "client_secret": "<YOUR SERVICE PRINCIPAL CLIENT SECRET>",
    "subscription_id": "<YOUR AZURE SUBSCRIPTION ID>",
    "resource_group_name": "<RESOURCE GROUP WITH YOUR ADF INSTANCE>",
    "factory_name": "<YOUR ADF FACTORY NAME>",
}

# Create the definition store
factory_definition_store = FactoryDefinitionStore(**factory_options)
```

<Admonition type="warning" title="Data Factory permissions">
Your service principal must have permissions to read your Data Factory resource. The Data Factory Contributor role provides adequate permissions.
</Admonition>

## Load and translate an ADF pipeline

Use the `FactoryDefinitionStore` to load an ADF pipeline by name. This will invoke client methods to:

- Fetch the pipeline definition.
- Fetch any required dataset, linked service, or trigger definitions.
- Translate the pipeline definition into an *internal representation* which can be used to create Databricks jobs.

```python
# Load and translate the ADF pipeline into wkmigrate's internal representation
pipeline_name = "<YOUR ADF PIPELINE NAME>"
pipeline_ir = factory_definition_store.load(pipeline_name)
print(pipeline_ir)
```

## Configure a Workspace Definition Store

Use a `WorkspaceDefinitionStore` to materialize the translated pipeline as a Databricks Lakeflow job. The `WorkspaceDefinitionStore` handles several migration steps:

- Emitting files detailing the job configuration, any required notebooks to be generated, and any untranslated activities.
- Uploading notebooks and supporting artifacts.
- Creating or updating Databricks Jobs (workflows).

Authentication is supported using a personal access token (PAT), username and password, or Azure client secret.

```python
from wkmigrate.definition_stores.workspace_definition_store import WorkspaceDefinitionStore

# Specify credentials and service details of the Databricks workspace
workspace_options = {
    "host_name": "https://adb-<workspace-id>.<region>.azuredatabricks.net",
    "pat": "<DATABRICKS PAT>",
    "authentication_type": "pat",
}

# Create the definition store
workspace_definition_store = WorkspaceDefinitionStore(**workspace_options)
```

<Admonition type="warning" title="Workspace permissions">
The credentials you provide must have permission to create jobs, upload notebooks, and manage secrets in the target Databricks workspace.
</Admonition>

## Translate pipelines to Databricks Lakeflow Jobs

The `WorkspaceDefinitionStore` materializes Databricks Lakeflow Jobs either by creating Databricks Asset Bundles or directly calling Databricks APIs to create jobs. 

### Create Databricks Asset Bundles
If you want to inspect or customize the translated workflow before deploying it to Databricks, use ``to_asset_bundle`` to create a deployable [Databricks Asset Bundle](https://docs.databricks.com/aws/en/dev-tools/bundles/).

```python
from pathlib import Path
from wkmigrate.models.ir.pipeline import Pipeline

# Directory where generated notebooks, job definitions, and other metadata will be written
local_dir = Path("/wkmigrate_pipeline/bundle")
local_dir.mkdir(parents=True, exist_ok=True)

# Create job artifacts from the pipeline's internal representation
workspace_definition_store.to_asset_bundle(pipeline_ir, bundle_directory=str(local_dir))
```

### Create Databricks Lakeflow Jobs directly

Once you're ready to deploy, use `to_job` to create a Databricks Lakeflow Job directly from the translated pipeline. This will:

1. Upload any required notebooks into your workspace.
2. Create any required Delta Live Table pipelines.
3. Calls `WorkspaceClient.jobs.create(...)` to register the job using properties translated from Azure Data Factory.

```python
# Create a Databricks Lakeflow Job (workflow) from the translated pipeline
job_id = workspace_definition_store.to_job(pipeline_ir)

print(f"Created Databricks job with ID: {job_id}")
```

<Admonition type="warning" title="Idempotency and naming">
Repeatedly calling `to_job` may create multiple identical jobs unless you implement your own idempotency strategy. Consider cleaning up test jobs manually while you iterate.
</Admonition>

## Translate multiple pipelines

When migrating an entire Data Factory, you can use the batch methods to translate and deploy all pipelines in a single run. Individual pipeline failures are logged and skipped so that one broken pipeline does not prevent the rest from being translated.

### List and load all pipelines

Use `list_pipelines` to list every pipeline in the data factory resource, then `load_all` to translate them in bulk. You can optionally pass a list of pipeline names to translate a subset.

```python
# List all pipeline names in the factory
pipeline_names = factory_definition_store.list_pipelines()
print(f"Found {len(pipeline_names)} pipelines: {pipeline_names}")

# Translate all pipelines (failures are logged and skipped)
all_pipelines = factory_definition_store.load_all()

# Or translate a specific subset
selected_pipelines = factory_definition_store.load_all(
    pipeline_names=["pipeline_a", "pipeline_b"]
)
```

### Create Databricks Asset Bundles for multiple pipelines

Use `to_asset_bundles` to generate separate [Databricks Asset Bundle](https://docs.databricks.com/aws/en/dev-tools/bundles/) subdirectories for each translated pipeline. Each bundle is written to a subdirectory named after the pipeline inside the parent directory you specify.

```python
from pathlib import Path

bundle_dir = Path("/wkmigrate_pipeline/bundles")
bundle_dir.mkdir(parents=True, exist_ok=True)

# Generate asset bundles for all translated pipelines
workspace_definition_store.to_asset_bundles(all_pipelines, bundle_directory=str(bundle_dir))
```

### Create Databricks Lakeflow Jobs for multiple pipelines

Use `to_jobs` to deploy translated pipelines as Databricks Lakeflow Jobs. The method returns a list of job IDs for the successfully created jobs.

```python
# Deploy all translated pipelines as Databricks jobs
job_ids = workspace_definition_store.to_jobs(all_pipelines)

print(f"Created {len(job_ids)} jobs: {job_ids}")
```

<Admonition type="info" title="Error handling in batch methods">
When using `load_all`, `to_jobs`, and `to_asset_bundles` failures of a single pipeline will raise a warning and skip translation of the failed pipeline. The remaining pipelines will be processed. Check logs for warnings after a batch run to identify any failures.
</Admonition>

## Customize translation with definition store options

`WorkspaceDefinitionStore` accepts an `options` dictionary that customizes how workflows are generated and deployed. 
Specifying definition store options will adapt the migration output without modifying the translated pipeline definition.

### Available options

| Option                 | Type                          | Description                                                                                                                                                     |
|------------------------|-------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `workspace_url`        | `str`                         | Overrides the Databricks host URL used when resolving linked services during translation.                                                                       |
| `root_path`            | `str`                         | Rewrites notebook and pipeline paths so artifacts are placed under a different workspace directory. Useful for cross-workspace or cross-environment migrations. |
| `compute_type`         | `"serverless"` or `"classic"` | When set to `"serverless"`, strips `new_cluster` definitions so that jobs run on serverless compute.                                                            |
| `files_to_delta_sinks` | `bool`                        | Converts non-Delta file sinks to Delta format during DLT pipeline generation.                                                                                   |
| `catalog`              | `str`                         | Overrides the Unity Catalog catalog name used for generated DLT pipelines.                                                                                      |
| `schema`               | `str`                         | Overrides the Unity Catalog schema name used for generated DLT pipelines.                                                                                       |

### Set options at construction time

Pass an `options` dictionary when you create the definition store:

```python
workspace_definition_store = WorkspaceDefinitionStore(
    host_name="https://adb-<workspace-id>.<region>.azuredatabricks.net",
    pat="<DATABRICKS PAT>",
    authentication_type="pat",
    options={
        "root_path": "/Workspace/Users/team@example.com/migrated",
        "compute_type": "serverless",
        "catalog": "main",
        "schema": "bronze",
    },
)
```

### Set options after construction

You can also modify options on an existing store using `set_option` and `set_options`:

```python
# Set a single option
workspace_definition_store.set_option("compute_type", "serverless")

# Replace all options at once
workspace_definition_store.set_options({
    "root_path": "/Workspace/Shared/prod",
    "files_to_delta_sinks": True,
})
```

Options are stored in the `options` attribute and can be read directly:

```python
print(workspace_definition_store.options)
# {"root_path": "/Workspace/Shared/prod", "files_to_delta_sinks": True}
```

### Example: Deploying to a different workspace path with serverless compute

```python
workspace_definition_store = WorkspaceDefinitionStore(
    host_name="https://adb-<workspace-id>.<region>.azuredatabricks.net",
    pat="<DATABRICKS PAT>",
    authentication_type="pat",
    options={
        "root_path": "/Workspace/Shared/production/pipelines",
        "compute_type": "serverless",
    },
)

# Options are applied automatically when creating jobs or asset bundles
job_id = workspace_definition_store.to_job(pipeline_ir)
```
